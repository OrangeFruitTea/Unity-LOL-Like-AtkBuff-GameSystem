# 网络同步服务脚本接口设计文档

| 属性 | 说明 |
|------|------|
| 文档类型 | API / 脚本接口契约（Interface Design）；**非** Mirror API 替代品 |
| 版本 | 1.0 |
| 上级设计 | 《[NetworkSyncService.md](./NetworkSyncService.md)》v2.x（Mirror、分层、服务端权威） |
| 允许的工程依赖 | `Assets/_Project/Code/Scripts/Basement` 内既有类型与既有模式 |
| Mirror 边界 | Mirror 的类型（`NetworkManager`、`NetworkBehaviour`、`[Command]` 等）仅在 **适配实现层**出现；外部业务**只见本文件定义的 façade 接口** |

---

## 1. 编写目的与读者

- **目的**：为「Gameplay、UI、工具、存档/配置加载」等与网络无关的系统提供**稳定的 C# 接口面**，避免因直接调用 `Mirror` 静态入口或零散 `FindObjectOfType` 导致的耦合；同时明确如何从 **Basement** 事件、JSON、对局时间上接入网络会话。
- **读者**：Gameplay 程序员、前端 UI、毕业设计答辩中阐述「分层与接口」时使用。

---

## 2. 设计原则（须遵守）

| 原则 | 说明 |
|------|------|
| 依赖倒置 | `Gameplay.*`、`UI.*` 依赖本节 **抽象接口**，由 `_Project/Code/Scripts` 下某 **Network/Facade 实现程序集**绑定 Mirror |
| 单一路径访问 Mirror | 禁止在项目内另起一套 socket/netcode；会话启停、`ServerChangeScene` 等须经 façade 或直接集中在一个 `NetworkManager` 子类中（与《NetworkSyncService》一致） |
| Basement 只做「横向能力」 | 事件总线、JSON、对局时间**不包含** Mirror 语义；网络层发布/订阅 Basement 事件时仅占位领域事件数据结构 |
| 可测试 | 界面与单机逻辑可对 `INetworkSessionReadOnly` 等做 **stub**（离线模式） |

---

## 3. Basement 与本接口的关系

### 3.1 推荐使用的能力

| Basement 组件 / 命名空间 | 用途（与网络会话结合） |
|--------------------------|------------------------|
| `Basement.Events.GameEventBus`（`Singleton<GameEventBus>`） | 网络会话状态迁移时 **Publish** 领域事件；UI/Gameplay **Subscribe**，与 Mirror 解耦 |
| `Basement.Events.IGameEvent` | 本项目网络相关领域事件均需实现该接口（见 §6） |
| `Basement.Events.Unity.EventDispatcher` | 若存在该组件且 **未**启用 ECS Basement 泵，则由其 `Update` 泵送事件；否则会话发布仍有效，需注意 §3.3 |
| `Basement.Json.JsonManager`（`Basement.Utils.Singleton<JsonManager>`）、`DeserializeFromFilePath<T>`、`JsonReadResult<T>`、`JsonSerializerProfile.GameContent` | 读取 `StreamingAssets` 下的**局域网地址、端口、发现键**等静态配置 |
| `Basement.MatchTime.IMatchTime` / `MatchTimeService.Instance` | 仅在 **对局已开始且未暂停**（`IsMatchActive` / `IsMatchPaused`）时驱动与「局内需同步的模拟」相关的 Basement 计时任务（如与 `TimingTaskScheduler` 连用）；**断线/未开局**可走独立 UI 或服务状态机 |
| `Basement.Tasks.Unity.TaskDispatcher`、`Basement.MatchTime.MatchTimeUnityDriver` | 与 ECS 泵的协调见 §3.3 |
| `Basement.Runtime.BasementRuntimeBootstrap` + `BasementLoopRouting.MarkEcsPumpActive`（若启用） | ECS 接管泵送时，`EventDispatcher`/`TaskDispatcher` 的 Unity `Update` 可能不再推进 Basement 环路，网络 façade 若以「每帧轮询 Mirror」方式写，应挂在 **仍存在 `Update`** 的对象上或由 ECS 统一调度 |

### 3.2 不推荐误用的情况

| 做法 | 原因 |
|------|------|
| 在 Buff/技能 ECS 管线内直接 `NetworkServer.Send` | 破坏权威边界；应先由 `NetworkBehaviour`/`Command` 收口再调 Gameplay |
| 用具象 `NetworkManager.singleton` 散落于 UI（无接口） | 难以单元测试与切换离线原型 |

### 3.3 与 Basement 环路（ECS 泵 vs Unity Update）

- 当 `Basement.Runtime.BasementRuntimeOptions.UseEcsWorldBasementLoop == true` 且 `BasementPumpEcsSystem` 已注册：`EventDispatcher`、`TaskDispatcher` 内对 `BasementUnityPump.PumpGameEvents()` / `MatchTimeService.Tick*` 的 **Unity Update 可能被跳过**。  
- **对网络 façade 的含义**：若在 `Update` 里读写 Mirror 会话状态，**不要**假设「事件泵一定在 EventDispatcher.Update 执行」；应保证 façade 自己有驱动（例如在独立 MonoBehaviour、`NetworkManager`、或 ECS System 中单点泵送）。本节接口**不关心**你从哪一侧驱动，只做契约。

## 4. 外部系统划分与接入面

| 外部系统 | 主要依赖接口（§5） | 典型用途 |
|----------|-------------------|----------|
| **菜单 / UI（主界面、局域网列表）** | `INetworkSessionReadOnly`、`INetworkSessionControl` | 展示连接状态；按钮 StartHost / StartClient / Stop |
| **局内 HUD（仅在联机时对「延迟提示」占位）** | `INetworkSessionReadOnly`、`INetworkDiagnosticsReadOnly`（可选） | 只读徽章，不写权威状态 |
| **Gameplay（技能、商店、胜负）** | `INetworkRoleReadOnly`、`IAuthoritativeActionGate`、`INetworkGameplayEvents`（或通过 GameEventBus 订阅 §6） | 判断服务端/主机、是否允许施加指令 |
| **配置与本地化工具** | 仅 `Basement.Json` + **本文件的 `INetworkConfigProvider`**（若实现） | 读表，不触碰 Mirror |

## 5. 核心脚本接口契约

> 以下为 **设计契约**：命名空间建议为 `Gameplay.Network`（或项目统一前缀）；可在实现时再拆分到 `Gameplay.Network.Abstractions`。  
> 方法签名以实现为准，本文固定**语义**，避免与 Mirror API 字面重名耦合。

### 5.1 会话只读：`INetworkSessionReadOnly`

**职责**：屏蔽 Mirror 的细节，向外提供「是否在局、宿主/服务端/客户端」等只读快照（供 UI 与Gameplay 分支）。

```csharp
namespace Gameplay.Network
{
    /// <summary> 当前进程的网络会话快照（可被轮询或由实现方通过事件触发 UI 刷新）。 </summary>
    public interface INetworkSessionReadOnly
    {
        bool IsHostRunning { get; }   // Host：同时 Server + Local Client 的典型场景
        bool IsServerActive { get; }
        bool IsClientConnected { get; }

        /// <summary> 是否已载入「网络游戏场景」（或由项目定义的 onlineScene 等价物）。可选，未判定时可为 false。 </summary>
        bool IsOnlineSceneLoaded { get; }

        ushort ActivePort { get; }
        string LastRemoteAddressOrEmpty { get; }
    }
}
```

**外部如何使用**

- UI 在每帧、`OnGUI`、或 façade 给出的 `event Action Changed`（实现可选）刷新按钮灰态。
- **禁止**仅靠该接口执行「服务端写权威生命值」等业务；只读分流。

---

### 5.2 会话控制：`INetworkSessionControl`

**职责**：集中 Host / Dedicated Server / Client 启动与停机，对齐《NetworkSyncService》中 NetworkManager 角色。

```csharp
namespace Gameplay.Network
{
    public enum NetworkStartResult
    {
        Success,
        AlreadyRunning,
        InvalidParameters,
        TransportError,
        AbortedByUser,
    }

    /// <summary> 供菜单层调用；底层对应 Mirror Transport + NetworkManager 子类。 </summary>
    public interface INetworkSessionControl : INetworkSessionReadOnly
    {
        NetworkStartResult TryStartHost(ushort listenPort);
        NetworkStartResult TryStartServer(ushort listenPort);
        NetworkStartResult TryStartClient(string host, ushort port);

        /// <summary> 幂等停机；若在局中应触发领域事件 §6「会话结束」。 </summary>
        void StopAll();

        /// <summary> 由<strong>服务端</strong>驱动切场景（对齐 Mirror ServerChangeScene 语义）；客户端禁止直接 LoadSceneAsync 顶替。 </summary>
        bool ServerChangeGameplayScene(string sceneName);
    }
}
```

**外部如何使用**

- **仅菜单/会话层**调用；Gameplay 不因「放一个技能」而调用 `TryStartHost`。
- `ServerChangeGameplayScene` 仅由**已处于 Server 角色**的代码路径调用（实现内应 `NetworkServer.active` 校验）。

---

### 5.3 角色与权威门控：`INetworkRoleReadOnly` + `IAuthoritativeActionGate`

**职责**：Gameplay 在发出「改变世界」的请求前询问「当前是否服务端权威」或「是否允许发往服务端」。

```csharp
namespace Gameplay.Network
{
    /// <summary> 不包含 Mirror API；等价于语义上的 Server/Host/LocalAuth。 </summary>
    public interface INetworkRoleReadOnly
    {
        bool HasServerAuthorityProcess { get; }
        bool IsClientOnlyProcess { get; }
    }

    /// <summary>
    /// Gameplay 侧的「闸门」：例如技能释放、商店购买应先经此判断；
    /// 单机模式可固定返回 true 并由 façade 设为「本地即权威」。
    /// </summary>
    public interface IAuthoritativeActionGate : INetworkRoleReadOnly
    {
        /// <summary> 当前上下文是否应由本进程<strong>服务端</strong>直接执行游戏规则。 </summary>
        bool ShouldExecuteGameplayAuthoritativelyLocally { get; }

        /// <summary> 若 ShouldExecuteGameplayAuthoritativelyLocally 为 false，是否仍可发送意图到远端（一般由 Command 实现）。单机可 false。 </summary>
        bool CanSendGameplayIntentToServer { get; }
    }
}
```

**外部如何使用**

- **技能 / 商店 / Buff 挂载**：服务端路径直接调用现有 ECS/`BuffApplyService`；客户端路径改为 `Command`/消息，`IAuthoritativeActionGate` 只负责**不写错侧**。
- 与《[MOBA局内装备与商店模块设计文档](./MOBA局内装备与商店模块设计文档.md)》中「服务端权威」一致：购买意图经 façade → Mirror → 服务端校验。

### 5.4 局域网发现 façade（可选 · MVP 可不实现接口而直接 UI 写 IP）

```csharp
namespace Gameplay.Network
{
    /// <summary> 对应 Mirror NetworkDiscovery；用于 UI 展示可用主机列表或直连。 </summary>
    public interface ILanLobbyBrowser
    {
        void BeginListenForHosts();
        void StopListenForHosts();

        System.Collections.Generic.IReadOnlyList<LanHostAdvertisement> VisibleHosts { get; }
        event System.Action HostListChanged;

        void RequestConnect(in LanHostAdvertisement host);

        sealed class LanHostAdvertisement
        {
            public string EndPointHint;
            public ushort Port;
            /// <summary> UDP 载荷或自定义摘要，内容由实现映射 NetworkDiscovery。 </summary>
            public string SessionKeyOrEmpty;
        }
    }
}
```

**外部如何使用**

- 仅 **局域网大厅 UI** 依赖该接口；局内Gameplay不依赖列表，只依赖已建立的 `INetworkSessionReadOnly`。

### 5.6 会话配置：`INetworkConfigProvider`（基于 Basement.Json）

**职责**：从 `StreamingAssets` 读取默认值（地址、端口、发现键、`schemaVersion`），与 `Gameplay`/`SkillData` 同款 JSON 流水线。

```csharp
namespace Gameplay.Network
{
    /// <summary> DTO 由项目定义；建议使用 Newtonsoft 特性对齐 JSON，见 Basement Json 模块。 </summary>
    public interface INetworkConfigProvider
    {
        bool TryReload(out string error);

        Basement.Json.JsonReadResult<NetworkConfigDto> Current { get; }
    }

    /// <summary> 示例形状；最终以项目 JSON 为准。 </summary>
    public sealed class NetworkConfigDto
    {
        public int SchemaVersion;
        public ushort DefaultListenPort = 7777;
        public string FallbackBindAddressOrEmpty = "";
        public string DiscoveryKeyOrEmpty = "";
    }
}
```

**实现备注**：装载路径使用 `JsonManager.Instance.DeserializeFromFilePath<T>(路径, JsonSerializerProfile.GameContent)`（参见 `Basement.Json` 命名空间），错误处理对齐 `JsonReadResult` 的既有模式。

### 5.7 解析器入口聚合（ façade 提供者 / Service Locator 轻量版）

单机与联机可共用同一门面；推荐通过**单点**解析（任选一种实现即可，本节不强制 DI 容器）：

```csharp
namespace Gameplay.Network
{
    /// <summary> UI/Gameplay 静态入口；实现可由 NetworkManager Awake 时注册或由 ScriptableObject 赋值。 </summary>
    public static class NetworkFacades
    {
        public static INetworkSessionReadOnly Session { get; set; }
        public static INetworkSessionControl SessionControl => Session as INetworkSessionControl;

        public static IAuthoritativeActionGate Authority { get; set; }

        /// <summary> 可选；无外网或未加载 discovery 时为 null。 </summary>
        public static ILanLobbyBrowser Lobby { get; set; }

        public static INetworkConfigProvider NetConfig { get; set; }
    }
}
```

**外部如何使用**

- 启动阶段：**网络模块初始化**赋值 `NetworkFacades.*`。  
- 单元测试：**替换 stub**（`Authority.ShouldExecuteGameplayAuthoritativelyLocally == true`，`Session` 离线快照）。

## 6. Basement 事件契约（推荐实现的领域事件）

网络层在经历「连接建立 / 断开 / 服务端场景切换就绪」时可 **Publish**，供 UI/HUD/Basement 任务订阅。以下为 **建议的事件类型**，需在本项目中实现 `Basement.Events.IGameEvent`：

| 事件（建议命名） | 触发时机 | 订阅方示例 |
|------------------|----------|------------|
| `NetworkSessionStartedEvent`（含模式 Host/Server/Client） | `TryStart*` 成功后 | HUD、存档 |
| `NetworkSessionStoppingEvent` / `Stopped` | `StopAll`、断线检测到 | 停定时器、清空预测缓存 |
| `NetworkPlayerTopologyChangedEvent`（人数、房主） | Spawn/Disconnect 玩家 | Room UI |
| `NetworkGameplaySceneChangedEvent` | 服务端切换场景并完成加载 | `MatchTimeService`、关卡初始化（若与网络绑定） |

**发布示例（语义）**

```csharp
GameEventBus.Instance.Publish(new NetworkSessionStartedEvent { Mode = NetworkRunMode.Host });
```

**外部如何使用**

- **UI**：`GameEventBus.Instance.Subscribe<NetworkSessionStartedEvent>(handler)`；卸载时 **Unsubscribe**。
- **Gameplay**：对「断线后立即禁止输入」可同时监听 `Stopping` + 自身 `PlayerInput.disable`。

## 7. 与对局时间 `IMatchTime` 的配合

| 场景 | 建议 |
|------|------|
| 会话已连接但尚未 `BeginMatch` | 不向玩家展示依赖 `MatchElapsed` 的排名；或保持 `TimingTaskScheduler` 不注册局内需网络同步的任务 |
| 仅主机开始局 | 在服务端成功 **ServerChangeGameplayScene** 且规则允许开局时调用 `MatchTimeService.Instance.BeginMatch()`（或通过已有 `Basement.MatchTime.IMatchTimeControl` 等价物）——与现有 `MatchTimeUnityDriver`、`TaskDispatcher` 流程一致 |

**约束**：本篇不规定「谁先网络 Ready 谁先 BeginMatch」，由实现与游戏规则约定；接口层只要求 **Gameplay 不因网络层未泵送而导致双端时间尺度不一致**时在文档中自检。

## 8. 实现侧职责映射（简短）

| 接口 | 典型实现宿主 |
|------|----------------|
| `INetworkSessionControl` | Mirror `NetworkManager` 的子类脚本（或由子类组合的 `NetworkFacades` setter） |
| `IAuthoritativeActionGate` | 同上或通过 `NetworkServer.active`、`NetworkClient.active` 封装的 Pure C# façade |
| `ILanLobbyBrowser` | `NetworkDiscovery` / `Telepathy`/KCP 参数包装 |
| `INetworkConfigProvider` | MonoBehaviour，`JsonManager` 读 `StreamingAssets`**网络配置 JSON** |
| Basement 事件 | 在具体 `override OnStartServer`、`OnStopClient`、断线回调中 Publish |
