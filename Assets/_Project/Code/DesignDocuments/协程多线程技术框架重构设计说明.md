# 协程多线程技术框架重构设计说明

## 1. 重构背景

为了解决协程设计模块中可依赖任务与简单任务区分带来的复杂性，提高代码的可维护性和扩展性，我们对协程多线程技术框架进行了重构。

## 2. 核心设计变更

### 2.1 统一Task组件设计

- **取消可依赖任务与简单任务的区分**：移除了`DependentTask`和`ITaskDependency`接口，统一使用`ITask`接口
- **自动归类简单任务**：所有依赖为空的任务自动视为简单任务
- **简化任务创建流程**：通过`ActionTask`直接包装Action委托，减少中间层

### 2.2 接口设计

**ITask接口**：
- 添加了`Status`属性，用于表示任务状态
- 添加了`Dependencies`和`Dependents`属性，用于管理任务依赖关系
- 添加了`Exception`属性，用于存储任务执行异常
- 添加了`AddDependency`和`RemoveDependency`方法，用于管理依赖
- 添加了`UpdateStatus`方法，用于更新任务状态
- 添加了`OnStatusChanged`回调，用于通知状态变化

### 2.3 实现类设计

**BaseTask抽象类**：
- 实现了`ITask`接口的核心功能
- 提供了线程安全的依赖管理
- 实现了任务状态的自动更新逻辑
- 实现了任务判等方法（基于Id）

**ActionTask类**：
- 继承自`BaseTask`
- 用于包装Action委托作为任务执行
- 提供了异常处理和状态管理

### 2.4 系统架构调整

- **移除TaskDependencyManager**：依赖关系直接由任务自身管理
- **移除TaskDependencyVisualizer**：简化系统结构
- **调整TaskScheduler**：使用新的ITask接口，支持直接创建和提交任务
- **调整ThreadingSystem**：移除了旧的任务创建方法，提供统一的任务创建接口

## 3. 技术实现

### 3.1 任务依赖管理

- **依赖添加**：通过`AddDependency`方法添加依赖任务，自动更新状态
- **依赖移除**：通过`RemoveDependency`方法移除依赖任务，自动更新状态
- **状态计算**：当所有依赖任务完成时，任务自动变为Ready状态
- **循环依赖检测**：在添加依赖时检测是否存在循环依赖

### 3.2 任务调度

- **优先级调度**：根据任务优先级进行调度
- **状态管理**：根据任务状态进行生命周期管理
- **并发安全**：使用锁机制确保线程安全

### 3.3 错误处理

- **异常捕获**：在任务执行过程中捕获异常
- **异常存储**：将异常存储在任务的`Exception`属性中
- **状态更新**：异常发生时自动将任务状态设置为Failed

## 4. 代码结构

### 4.1 核心文件

- **ITask.cs**：任务接口定义
- **BaseTask.cs**：基础任务实现
- **ActionTask.cs**：Action任务实现
- **TaskScheduler.cs**：任务调度器
- **ThreadPool.cs**：线程池
- **WorkerThread.cs**：工作线程
- **ThreadingSystem.cs**：协程多线程系统主入口
- **CoroutineManager.cs**：协程管理器

### 4.2 移除文件

- **DependentTask.cs**：已被BaseTask替代
- **ITaskDependency.cs**：已被ITask替代
- **TaskDependencyManager.cs**：依赖管理功能已集成到BaseTask
- **TaskDependencyVisualizer.cs**：简化系统结构，移除可视化功能

## 5. 使用示例

### 5.1 创建简单任务

```csharp
// 创建并提交简单任务
var task = threadingSystem.CreateTask(() => {
    Debug.Log("简单任务执行");
});
threadingSystem.SubmitTask(task);
```

### 5.2 创建带依赖的任务

```csharp
// 创建任务A
var taskA = threadingSystem.CreateTask(() => {
    Debug.Log("任务A执行");
});

// 创建任务B，依赖于任务A
var taskB = threadingSystem.CreateTask(() => {
    Debug.Log("任务B执行");
});

// 添加依赖关系
taskB.AddDependency(taskA);

// 提交任务
threadingSystem.SubmitTask(taskA);
threadingSystem.SubmitTask(taskB);
```

## 6. 优势与改进

### 6.1 优势

- **简化设计**：取消了可依赖任务与简单任务的区分，减少了代码复杂度
- **提高可维护性**：统一的任务接口和实现，便于理解和维护
- **增强扩展性**：基于BaseTask的设计，便于添加新的任务类型
- **提高性能**：减少了中间层，提高了任务执行效率

### 6.2 改进空间

- **任务可视化**：可以考虑重新实现任务依赖关系的可视化功能
- **任务监控**：可以添加任务执行时间、资源使用等监控功能
- **任务重试**：可以添加任务失败后的重试机制

## 7. 总结

通过这次重构，我们成功地统一了Task组件设计，取消了可依赖任务与简单任务的区分，将所有依赖为空的任务自动归类为简单任务。重构后的系统更加简洁、高效，具有更好的可维护性和扩展性。

系统现在能够更灵活地处理任务依赖关系，同时保持了代码的清晰性和可读性。这种设计不仅满足了当前的需求，也为未来的功能扩展奠定了良好的基础。
