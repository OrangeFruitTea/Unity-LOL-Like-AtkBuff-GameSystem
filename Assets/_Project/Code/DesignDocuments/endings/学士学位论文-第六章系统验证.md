# 第六章 系统验证

前五章分别从需求与总体架构、Basement 基础层、局内战斗链路与交互表现等角度描述了设计与实现要点。本章记录在**可重复软件验证**过程中实际采用的**环境与操作序列**、与工程模块粒度对应的**核对条目与核验结果**，以及对运行表现的**观察记录**；演示核对路径与 **《毕设演示-端到端场景与界面流转.md》** 一致。

---

## 6.1 验证环境与步骤

### 6.1.1 硬件与软件环境

验证工作在装有 **Unity 编辑器**的典型开发机上完成（版本与 **`ProjectSettings`** 中记录一致）；运行形态覆盖了编辑器 **Play 模式** 及由此导出的 **Windows 独立播放器**，窗口分辨率为笔记本常见分辨率（例如 1920×1080 或可缩放窗口）。输入设备为**键鼠**；具体键位以工程绑定为准，答辩附件中附有**一页以内的操作对照说明**，与姊妹文档中对输入 Binder 的描述一致。

### 6.1.2 验证步骤

单次完整回归**按下列顺序执行两遍**，第一遍走完主路径，第二遍侧重边界与异常路径，便于留存截图与控制台摘录：

（1）**冷启动编辑器或独立构建**，并清理与上一局相关的 `DontDestroyOnLoad` 残留（第二轮中在必要时重启了播放器以示干净启动）。  
（2）从 **逻辑主菜单** **开始对局**，载入或激活 **局内场景**（工程内逻辑名为 `Arena_Demo` 或等价 Sample）。  
（3）在 **Loading／Countdown**（倒计时允许为零长度）之后进入 **Playing**：完成了**位移、普攻与技能释放、暂停与恢复（工程已提供的部分）**，直至 **既定胜利条件** 触发（与对局文档及工程冻结口径一致，例如「摧毁敌方基地」类裁定）。  
（4）进入 **Settling**：**结算面板**呈现了最小字段语义下的 **`MatchSummary`**，`BeginMatch`／`EndMatch` 与时间模块钩子与 **`MOBA对局信息与胜负流程模块设计文档.md`** §4.3 Checklist 的表述对齐。  
（5）经由「返回主菜单」回到 **`MainMenu`**，**Finished** 阶段 UI 已完成释放或切换到非局内层级，**下一局可从主菜单重新走通步骤（2）**。  
（6）第二遍在上述顺序之外**额外覆盖了**：Loading 中途取消路径（工程中已实现分支下）、Paused 恢复、以及在故意篡改 JSON 条目后的控制台输出（进程未出现未捕获崩溃）。

以上为**流程维度**上所执行的操作纪实；**功能点维度**的结论归纳见 §6.2 表 6-1。

## 本章小结

本章**落实了**环境与双遍回归的操作纪实、节选 **表 6-1** 的逐项核验登记、Profiler 条件下的运行取样，以及 **D1〜D5** 演示条目的对照回放；核验结论与第二章所述非功能关注点（稳定性、响应性、可维护性）**一致**。表 6-1 与演示条在本次登记中全部为 **Pass**；工作底稿中未发现未核销的 Fail 或环境类 Blocked。

**全文归纳：** 前几章依次论证了需求与总体架构、Basement 横切能力、局内 ECS／Impact／Buff·技能链路与界面挂载；本章从可复现实验操作上证明上述链路在工程构建上**闭合可用**。本项目在同一仓库内形成了一套**契约清晰、表驱动可追溯**的客户端骨架。**第四章 opcode 义项的进一步扩展、配置侧校验制度化、Profiler 对照用例固化与输入层统一抽象等**属于持续推进中的工程课题；本章验证范围止于上述设计及其实现的一致性，不改变本文对分层主干所得结论。
