# 相关知识简介：Unity 引擎与工作方式概览

> 本文稿在装订目录中通常为**第二章**，位于「第一章　绪论」之后、「第三章　需求分析与总体设计方案」之前，用文字做**鸟瞰式铺垫**。读者若尚未使用过 Unity，可先建立对编辑器与运行时的大致图景；专有名词够用即可，后文 Basement、Gameplay 与设计文档中会再逐项落到类名与职责。

---

## Unity 大致是什么

**Unity** 是一套用于做**三维／二维交互式实时内容**的软件：既包含在电脑上安装的 **编辑器**（做场景布置、预览、导出），也包含**打包后**在用户机器或手持设备里运行的**运行时 Player**。做游戏、可视化仿真、虚拟现实演示等都可以用同一套流水线完成；本课题的俯视角 MOBA 原型就属于「在编辑器里拼装场景与逻辑，再打 Windows 等平台可执行程序」这一种典型用法。

可以理解为：**编辑器负责「看得见、拖得动的创作界面」**，**运行时负责「按帧循环把画面和操作跑起来」**。两者共用同一套资源（模型、图片、音效、脚本等），但工作内容不同——前者偏制作与调试，后者偏稳定播放与性能边界。

---

## 编辑器一侧：关卡、预制体与「搭积木」

在 Unity 里，一个可玩的关卡一般由 **Scene（场景）** 来描述。场景里是许多 **GameObject（游戏物体）**：像舞台里的摆件，每个物体可以挂上若干 **Component（组件）**——例如决定在哪里的 **Transform**，决定受不受物理影响的 **Collider／Rigidbody**，播放动画用的 **Animator**，以及我们写的 **脚本（一般用 C#）**。

经常复用的一段组合（例如「带血条小兵」）会做成 **Prefab（预制体）**：像保存好的样板间，场景中拖出来的是它的**实例**。改 Prefab 会连带影响所有实例，适合做批量迭代；这也正是工程里大量使用预制体的原因之一。

编辑器里还能看到 **Hierarchy（层级）／Inspector（检视）／Project（资源）／Scene／Game** 等窗口：一边管场景树，一边改选中物体属性，一边浏览工程文件夹里的资源。**Play 按钮**按下后会在编辑态里起一个临时运行会话，停了又还原场景——适合快速试错，但要注意「Play 改掉的数据有时不会保存」这一类习惯性问题。

至此可以记住一个简单的画面：**关卡 = 场景；物体 = GameObject；能力 = Component 拼装；反复复用 = Prefab**。后文谈的桥接 ECS、Impact、UI 挂载等，归根结底都是在这条「物体 + 组件 + 脚本」的主干上做扩展或拆层。

## 运行时一侧：帧循环、「系统」与文化上的模块划分

运行时可以粗想成：**每一帧刷新输入 → 逻辑更新 → 物理／动画／其它子系统协同 → 最后画一帧画面**。Unity 在用户脚本层面暴露 **`Update`、`FixedUpdate`、协程、`ScriptableObject`** 等不同节拍与数据容器，为的是让「想清楚什么时候改数据、什么时候画图」更简单。

从产品宣传与文档结构上，常会听到这些**大而化之的功能簇**——不必背定义，只需知道论文里偶尔会点到它们的名字：

- **渲染（Rendering）**：把几何、材质与光照送进图形 API，最终在屏幕成像；本课题俯视相机、UI 画布都跑在这条链路上。
- **物理（Physics）**：简化过的刚体碰撞、射线检测等（是否采用、用在哪一类交互上要看项目）。
- **动画（Animation / Animator）**：状态机与时间轴驱动的角色肢体与特效节拍；工程中 `Animator`、动画事件常与战斗节奏对齐。
- **音频（Audio）**：音源组件与监听器的三维或二维混音——若毕设音效从简也可以几乎不提。
- **UI（uGUI 等）**：画布、按钮、HUD；交互表现与用户界面一章所述的 `UIManager` 一脉即建于 Unity UI 的生态之上。
- **资源管线（Assets / Resources / StreamingAssets）**：资源如何进包体、如何在运行时按路径加载表与媒体；工程里常把只读配置表放在 `StreamingAssets` 一类目录，由后文所述数据管线读取。

这些内容在技术上彼此关联，编辑器里却往往以**可选包（Package）、菜单或窗口入口**拆开呈现；本科论文不需要按官方架构图死记，只要在叙述自己的模块时能说出「利用了引擎哪一类能力」「哪一块由项目脚本包掉」。

## 选型相关背景：ECS 范式与 Unity 内置子系统

后文第三章「关键技术选型」会**结合本仓库**写明引擎、脚本语言与各层实现取舍；**本章不立项目自拟模块**，只交待两类可查公开资料的背景：**一是 ECS 的一般范式**（不限定本项目某一具体实现）；二是若干 **Unity 开箱自带、手册可查的子系统**，便于和后文「在通用引擎能力之上做自定义玩法层」相呼应。

---

### ECS（Entity–Component–System）架构范式

**ECS**不是某一家公司的商标，而是一类在**实时仿真、大规模游戏**里逐渐流行起来的**数据组织与执行模型**名字。可以把它理解成三部分：

1. **Entity（实体）**  
   通常只是**轻量身份**（常见实现里是一个整数 ID 或句柄），不携带复杂行为。场景里「这一只兵、那一个塔」在逻辑层面先落到一个实体编号上。

2. **Component（组件）**  
   挂在实体上的是**一小块数据**：例如坐标、血量、阵营、当前目标。**倾向用结构体／纯数据类型描述**，少用「又大又全的怪物类」。同一种实体可以同时拥有多种组件；**组合优于深继承**——这一点与教科书里强调的「别为每种单位写一条巨长继承链」相契合。

3. **System（系统）**  
   **遍历**「满足某组件组合的实体集合」，在同一类逻辑里做同一件事：比如所有带「可移动 + 导航」的单位走寻路积分、所有带「可受伤」的实体消费伤害队列。系统在每帧（或固定时间步）**顺序执行**；**谁先谁后**会直接影响语义（例如先扣 Buff 再结算伤害），因此公开资料里常会讨论**系统排序**或**阶段（Phase）划分**。

与「每个游戏对象一个大脚本、里面又调 API 又算规则」相比，ECS 强调的是：**数据聚拢、逻辑按系统切开、批量处理**。在实体数量上来之后，这种模式有利于**缓存友好**（CPU 连续访问同类型组件数组）和**并行化**——是否采用 **Unity DOTS / Entities**、是否用 **Jobs + Burst**、或采用其它轻量宿主，属于**具体引擎与版本层面的工程决策**，此处从略。读者只需抓住：**ECS 是文献与工业界讨论里常见的一类「多实体、规则频繁」场景的组织方式**；后文若出现具体类名与节拍 API，应视为**某一种实现**对范式的落地，而不是范式的全部内涵。

### 与 ECS 常被对照、或并列出现的「引擎侧」参考内容

下面这些**不是我方工程自研框架**，而是 Unity 编辑器与运行时里**开箱即有、手册可查**的能力；与 ECS 范式**可以同时出现在同一课题里**（例如规则在 ECS，表现仍用 MonoBehaviour 与 Animator），属于「知道它们从哪来、到哪去查文档」即可的层次。

- **GameObject + Component（Unity 经典对象模型）**  
  前文「搭积木」一节已提过：层级、Prefab、脚本挂在 Inspector 上。这是 Unity 默认的 **EC（Entity–Component）**味道，但**不等同于**上一小节的 ECS——后者通常还要求**系统批量、数据布局更规整**。论文里常说「混合动力」，指的就是在**引擎惯用 Mono 层**与**自管或第三方 ECS 宿主**之间切边界。

- **NavMesh（导航网格）寻路**  
  Unity 内置 **NavMesh** 与 **NavMesh Agent** 一类工作流：在场景里烘焙可走区域，运行时由 Agent 沿网格求路径。俯视角位移、点击地面移动等教程与商业项目普遍采用这一套，属于**导航子系统**，与「单位规则在 ECS 里还是 Mono 里」正交。

- **Animator / Mecanim 状态机**  
  **Animator Controller**把动画片段组织成状态与转移条件，由引擎根据参数驱动；**动画事件（Animation Events）**可在时间轴上打钩回调到脚本，用于对齐「出手帧」等现象。这是 **Unity 动画管线**的公开部分，与战斗节奏联调时常常会碰到。

- **物理与射线（PhysX 封装）**  
  Unity 的 **Rigidbody、Collider、Physics.Raycast** 等，底层来自 **NVIDIA PhysX** 的集成封装。用来做碰撞、触发器、从相机或单位位置发射射线选目标等，属于**引擎物理查询层**；是否 heavy 使用依项目而定。

- **协程（Coroutine）与引擎节拍**  
  Unity 在 **MonoBehaviour** 上提供基于 **`IEnumerator` + `yield`** 的协程，把「等若干秒 / 等一帧再继续」写进顺序感强的流程里，而无需到处手写计时器。这是 **C# 生成器在 Unity 运行时的惯用集成**，教材与官方手册均有专章。

- **uGUI（Canvas / EventSystem）**  
  **Canvas、RectTransform、Graphic Raycaster** 等构成 Unity 传统的屏幕 UI 栈；事件路由经 **EventSystem**。HUD、血条、按钮等多数教学项目与商业 UI 都建立在这套**官方 UI 基础**之上（与 UI Toolkit 等较新方案并存）。

## 为什么说 Unity 适合做本课题一类的毕设底座

下面这些点偏重**学与做的成本**，不过分强调商业排名：

第一，**一软多用**：装好编辑器即可在同一个工程里预览、打断点、Profiler 采样、导出多平台。**改完代码按 Play**，比「写完再整条编译链」爽快许多，尤其适合毕设迭代。

第二，**C# 与组件思维门槛友好**：专业课里常见的面向对象、接口、事件等在 Unity 语境里都能找到落点；与「纯手工写窗口消息循环」的旧式客户端相比，**起步曲线更平缓**。

第三，**资源丰富**：官方手册、问答社区与教学视频数量大，遇到粉色材质、Prefab 断开、图层排序等细碎问题也**容易检索到解法**——这对一个人扛完整条流水线的本科生很重要。

第四，**性能与可调测**：内置 **Profiler、Frame Debugger** 等工具；毕设虽未要求商业级画质，但需要**说清楚「我确实看过一眼性能」**，系统验证一章的取样描述即建立在这一前提下。

总之：Unity **不是万能的**，上层仍然要自己设计分层、表格驱动和事件节拍；但就「在有限时间里做出可把玩、可写进论文的原型」而言，它是一个**性价比高、上下文统一**的起点——这也是正文选择它作为宿主的原因。
