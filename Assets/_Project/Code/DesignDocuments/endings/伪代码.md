## 4.2.1 事件总线与接口契约

### 事件数据载体（`IGameEvent`）

```
类型 MyGameplayEvent 实现 IGameEvent:
    属性 EventId: 字符串        // 唯一语义标识，如 类型名
    属性 Timestamp: 日期时间    // 发布时若未设可填当前时间
    属性 Priority: GameEventPriority  // Low / Normal / High
```

### 订阅与广播（`GameEventBus`）

```
单例 GameEventBus

过程 Initialize():
    若已初始化则返回
    置已初始化标记

过程 Subscribe<T>(handler, options?)，约束 T : IGameEvent:
    若 handler 为空则返回
    在字典 _eventHandlers[typeof(T)] 中追加一条记录:
        Handler = handler
        Priority = options.Priority
        IsAsync = options.IsAsync
        Filter = options.Filter
    对该 T 的处理器列表按 Priority 降序排序

过程 Unsubscribe<T>(handler):
    从 _eventHandlers[typeof(T)] 中移除与 handler 相等的项

过程 Publish<T>(eventData)，约束 T : IGameEvent:
    若 eventData 为空则返回
    若 Timestamp 未设置则填当前时间
    复制当前 T 对应的处理器列表快照
    对快照中每一项:
        若定义了 Filter 且 Filter(eventData) 为假则跳过
        若 IsAsync 为真则在线程池中调用 handler(eventData)
        否则当前线程同步调用 handler(eventData)

过程 PublishAsync<T>(eventData):
    在线程池中执行 Publish(eventData)
```

### 调度与节流（`GameEventScheduler`）

```
单例 GameEventScheduler

过程 Schedule(event: IGameEvent):
    若 event 为空则返回
    将 event 入优先级队列，键为 (int)event.Priority

过程 ScheduleDelayed(event, delaySeconds):
    在场景中找一个 MonoBehaviour 启动协程:
        等待 delaySeconds
        再调用 Schedule(event)

过程 ProcessQueue():
    若正在处理则返回
    若队列为空则返回
    置「正在处理」
    循环 至多 _maxConcurrentEvents 次:
        若队列为空则跳出
        取队首事件 ev
        调用 GameEventBus.Instance.Publish(ev)
    清除「正在处理」

过程 Update():  // 由 BasementUnityPump.PumpGameEvents 等每帧调用
    若 Time.time - _lastProcessTime < _processInterval 则返回
    调用 ProcessQueue()
    更新 _lastProcessTime
```

### 与 Unity 生命周期的桥接（`EventDispatcher`）

```
组件 EventDispatcher : MonoBehaviour

过程 Awake():
    GameEventBus.Instance.Initialize()
    GameEventScheduler.Instance.Initialize()

过程 Update():
    若 BasementLoopRouting.IsEcsPumpActive:
        返回   // ECS 侧由 BasementPumpEcsSystem 统一 Pump，此处避免重复
    调用 BasementUnityPump.PumpGameEvents()
        // 内部等价于 GameEventScheduler.Instance.Update()

过程 OnDestroy():
    GameEventBus.Instance.Clear()   // 清空全部订阅

静态过程 GetOrCreate():
    若场景中存在 EventDispatcher 则返回之
    否则 Create()：new GameObject("EventDispatcher") 并挂载组件后返回
```

### 发布方：立即广播 vs 先入队

```
过程 业务模块通知监听器(数据):
    // 路径 A：同步广播（多见于网络会话、购买结果等）
    GameEventBus.Instance.Publish( new SomeGameEvent(数据) )

    // 路径 B：先入队，在调度节拍内再 Publish（缓解同帧密集触发）
    GameEventScheduler.Instance.Schedule( new SomeGameEvent(数据) )
```

## 与「如何观测」的边界说明

**事件分发**解决模块间**松耦合通信**；**日志 / 诊断**（如 `LogManager`、Editor 下 `EventDebugWindow`）解决**可观测性**，二者职责独立。监听器可在 **Subscribe** 的 handler 内自行写日志或转发到调试窗口，但**不应**把日志实现写进 `GameEventBus` 核心路径，以保持 Basement 层抽象稳定。
