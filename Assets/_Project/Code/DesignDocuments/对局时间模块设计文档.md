# 对局时间（Match / Combat Time）模块设计文档

## 1. 文档目的与范围

本文档定义一套**简单、可复用**的**对局内时间**抽象，用于统一技能管线、Buff 衰减、延时任务、ECS 系统等对「时间流逝」的读法，减少 `Time.time`、`Time.fixedDeltaTime`、`DateTime.Now` 混用带来的**时序漂移与暂停行为不一致**。

- **范围**：单局对战从开始到结束（含暂停、重开）；不包含服务器权威时钟同步细节（可在网络文档中扩展）。
- **非目标**：不替代 Unity 的物理步进；不实现复杂时间曲线或子弹时间栈（可后续扩展）。
- **参考**：`Assets/_Project/Code/Scripts/Basement/TimingTask` 下 `TimingTaskScheduler` 当前以 `UnityEngine.Time.time` 计算任务间隔（见下节）。

---

## 2. 与现有 `TimingTask` 的关系

### 2.1 现状（代码事实）

| 组件 | 时间来源 | 说明 |
|------|----------|------|
| `TimingTaskScheduler.Update()` | `Time.time` 与 `_lastUpdateTime` 的差值作为 `deltaTime` | 调度器另有 `_updateInterval`（默认约 0.016s），实际推任务的时间粒度为「更新间隔 + 累计误差」。 |
| `TimingTaskBase` / `ITimingTask` | `DelayTime` 为秒，由调度器递减 | 语义为「游戏时间秒」，与调度器使用的 `deltaTime` 一致。 |

因此：**现有 TimingTask 已跟随 Unity 的 `Time.time` 刻度**（默认受 `Time.timeScale` 影响）。若全局 `timeScale = 0`（暂停），调度器在仍被 `Update` 调用的前提下，`Time.time` 不变，`deltaTime` 为 0，任务倒计时**会停滞**——这与多数「暂停停表」需求一致。

### 2.2 仍需要对局时间模块的原因

1. **显式边界**：对局开始/结束时应**重置**「逻辑时间」与统计，避免沿用上局 `Time.time` 的绝对值做相对计时产生歧义。
2. **单一读入口**：技能设计文档要求 **CombatClock / MatchTime**；Buff 使用 `FixedUpdate` 与 `Time.fixedDeltaTime`；Basement 任务使用 `Update` 与 `Time.time`。**统一文档与 API** 后，策划与程序约定「本模块读 `IMatchTime` 的哪几个量」。
3. **演进空间**：未来若需「仅暂停角色、不暂停 UI 动画」或「任务使用非缩放时间」，可在本模块内分层，而不必全项目替换 `Time` 调用。

## 3. 设计目标

1. **对局时间（Match Time）**：从 `MatchStart` 起算的单调递增逻辑时间（秒），暂停时可选择**不递增**或按独立策略递增（见 5.2）。
2. **统一增量**：提供 `DeltaTime`、`FixedDeltaTime`（或明确映射到 Unity 的 scaled / unscaled），供 ECS System、技能管线、与 Buff 侧**文档约定**使用。
3. **生命周期清晰**：`BeginMatch` → `Pause` / `Resume` → `EndMatch`；结束时可选清理与 `TimingTaskManager.ClearAllTasks()` 等联动（由集成方调用，本模块可发事件或回调）。
4. **实现简单**：首选**薄封装** Unity `Time` + 对局内 `elapsed` 累加，避免首版引入复杂依赖。

---

## 4. 核心概念

| 概念 | 含义 |
|------|------|
| **逻辑秒（Scaled）** | 与 `Time.timeScale` 一致的时间，用于技能 CD、Buff 持续时间、多数 TimingTask。 |
| **真实秒（Unscaled，可选）** | `Time.unscaledDeltaTime` 等，用于加载遮罩超时、断线重连提示等**不随暂停停表**的需求。 |
| **对局已进行时间 `MatchElapsed`** | 自本局 `BeginMatch` 起累计的逻辑秒，**暂停时不累计**（默认）。 |
| **固定步增量 `MatchFixedDeltaTime`** | 与 `Time.fixedDeltaTime` 对齐的 scaled 固定步长，供「必须与物理步一致」的逻辑阅读（如与现有 Buff 衰减对齐时的参考）。 |

## 5. 模块职责与 API 建议

### 5.1 接口草案（`IMatchTime` 或 `ICombatClock`）

仅列**最小只读面**（实现可为 `Singleton` 或由 DI 注入）：

```csharp
// 命名空间示例：Basement.Time 或 Core.Gameplay.Time
public interface IMatchTime
{
    /// <summary> 本局是否进行中。 </summary>
    bool IsMatchActive { get; }

    /// <summary> 自 BeginMatch 起累计的逻辑秒（暂停不增）。 </summary>
    float MatchElapsed { get; }

    /// <summary> 当前帧逻辑增量（通常等同 Time.deltaTime，受 timeScale 影响）。 </summary>
    float DeltaTime { get; }

    /// <summary> 当前固定步逻辑增量（通常等同 Time.fixedDeltaTime * 有效缩放）。 </summary>
    float FixedDeltaTime { get; }

    /// <summary> 与 Unity Time.time 对齐的全局逻辑时间快照（便于与现有 TimingTask 对照）。 </summary>
    float UnityScaledTime { get; }
}
```

### 5.2 控制面（`MatchTimeController` 或合并进同一服务）

```csharp
public interface IMatchTimeControl : IMatchTime
{
    void BeginMatch();
    void EndMatch();
    void PauseMatch();   // 可选：Time.timeScale=0 或仅内部 _paused 标志
    void ResumeMatch();
}
```

- **默认暂停策略**：`PauseMatch` 将内部 `_paused` 置 true，`MatchElapsed` 不再累加；若项目统一用 `timeScale=0`，则 `DeltaTime` 自然为 0，**二者可同时使用**，但需在团队内固定一种「主开关」，避免双重暂停逻辑冲突。
- **EndMatch**：重置 `MatchElapsed`、`_paused`；是否调用 `TimingTaskManager.ClearAllTasks()` 建议由**对局流程管理器**显式编排，本模块可提供 `OnMatchEnded` 供订阅。

### 5.3 更新方式

- **Update 阶段**：每帧 `MatchElapsed += DeltaTime`（当 `IsMatchActive && !_paused`）。
- **与 `TaskDispatcher` 的关系**：保持现有 `TaskDispatcher.Update` → `TimingTaskScheduler.Update()`；本模块可与 `TaskDispatcher` 挂在同对象或同帧序，**不要求**首版改动 `TimingTaskScheduler` 源码。长期可选：让调度器使用 `IMatchTime.DeltaTime` 替代自行计算 `Time.time - _lastUpdateTime`，以消除调度器 `_updateInterval` 与全局 `DeltaTime` 的细微差异。

## 6. 与各系统的使用约定

| 消费者 | 建议读法 | 说明 |
|--------|----------|------|
| **技能管线（ECS）** | `MatchElapsed` + `DeltaTime` 推进延迟步骤 | 与《基于Buff与ECS的技能系统设计文档》中「相对 CastStart」一致时，可存 `castStartMatchElapsed`。 |
| **BuffManager** | 短期可继续 `FixedUpdate` + `Time.fixedDeltaTime`；中期在文档中约定是否改为 `IMatchTime.FixedDeltaTime` 或仍跟物理 | 若暂停时 Buff 也应停表，需保证 `fixedDeltaTime` 在暂停时为 0 或 Buff 侧检查 `_paused`。 |
| **Basement TimingTask** | 保持现状即可；新任务逻辑若需「对局外仍走表」再引入 unscaled 路径 | 首版不强制改 `TimingTaskScheduler`。 |
| **Impact / 战斗结算** | 事件时间戳可记 `MatchElapsed` 便于回放对齐 | 可选。 |

## 7. 实现阶段建议

| 阶段 | 内容 |
|------|------|
| **MVP** | `MatchTimeService`：`BeginMatch`/`EndMatch`/`Pause`/`Resume`，`Update` 中维护 `MatchElapsed`；对外 `IMatchTime`；在入口场景或 `GameSession` 中驱动 `Update`。 |
| **集成** | 技能 `SkillCastPipelineSystem`、调试 HUD 显示对局时间；文档中替换散落的 `Time.time` 相对计时。 |
| **可选** | 将 `TimingTaskScheduler.ProcessTasks` 的 `deltaTime` 改为注入 `IMatchTime.DeltaTime`；提供 `UnscaledDeltaTime` 与独立累计 `RealElapsed`。 |

---

## 8. Unity 2022 注意点

- `Time.time`、`deltaTime` 受 `timeScale` 影响；`fixedDeltaTime` 可通过 `Time.fixedDeltaTime` 读取，暂停时 FixedUpdate 可能仍被调用（取决于 `maximumDeltaTime` 等），**Buff 若必须在暂停时完全停止**，应结合 `IMatchTime.IsMatchActive` / `_paused` 判断。
- 不在本模块内使用 `DateTime.Now` 驱动游戏逻辑（网络对时另议）。

---

## 9. 小结

对局时间模块是 **`Time` 的语义化薄层 + 本局 `MatchElapsed` 与暂停策略**，与 **`Basement.TimingTask` 自然兼容**（同基于 scaled 时间）；首版**不强制修改** `TimingTaskScheduler`，通过全项目约定 **战斗与技能读 `IMatchTime`** 即可落地《技能系统设计》中的统一时序目标，并为后续与 Buff 固定步、网络帧对齐预留扩展点。
